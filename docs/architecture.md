# Architecture

Breadcrumb sends an agent's structured experience report to infrastructure controlled by
the software owner. It does not require a hosted Breadcrumb analytics service.

## Data flow

```text
Any participating agent (Codex, Claude Code, browser agent, or another runtime)
  │ discovers /.well-known/breadcrumb, robots.txt, llms.txt, or HTML metadata
  │ reads owner policy, protocol, and report schema
  │ directly POSTs declared outcome, attribution, and friction; no install required
  ▼
POST owner submission_url
  ▼
@breadcrumb/server
  │ size check → JSON parse → schema and policy validation
  │ deduplication verification → rate limit → second sanitization
  ▼
BreadcrumbStorage
  ▼
@breadcrumb/postgres
```

## Package boundaries

### `@breadcrumb/core`

Owns protocol types, runtime JSON Schema validation, website and repository discovery,
privacy sanitization, stable deduplicating, and optional HTTP submission helpers. Direct
agents do not need to install or import it. It has no storage or server framework dependency.

### `@breadcrumb/server`

Owns the untrusted ingestion boundary. `createBreadcrumbHandler` accepts and returns Web
Fetch API objects, so adapters can expose it from frameworks that support `Request` and
`Response`. Storage is injected through the small `BreadcrumbStorage` interface.

### `@breadcrumb/postgres`

Implements `BreadcrumbStorage` with append-only JSONB records plus indexed columns for
common filtering and grouping. Agents never connect to PostgreSQL directly.

### `@breadcrumb/codex`

Provides an optional legacy runtime adapter for environments that want deterministic hook
metrics. It is not required for Codex—or any other agent—to discover and submit direct
feedback.

## Trust boundaries

1. **Direct agent input is untrusted.** Reports are schema and policy validated; supplied
   deduplication values are verified and omitted deduplication values are generated by the server.
2. **Owner metadata is untrusted.** Manifests are schema validated and network operations
   time out.
3. **Reports are untrusted.** The server validates independently, rejects private-content
   claims, recomputes deduplication, rate limits, and sanitizes again.
4. **Storage is owner-controlled.** The reference adapter receives only accepted reports.

## Failure behavior

Reporting is best effort and must not interfere with the agent's task. No meaningful
friction, missing manifests, network failures, and rejected event types result in no report.
A website cannot force an arbitrary agent to report; the agent must voluntarily follow the
published instructions.

## Current limitations

- Semantic friction, metrics, task outcome, attribution, and confidence are untrusted agent
  assertions rather than verified facts.
- The generic reference rate limiter is process-local; the Vercel adapter uses distributed
  PostgreSQL buckets.
- Bearer authentication requires externally provisioned credentials, which is incompatible
  with anonymous zero-install reporting. Signed agent identity is not implemented.
- Core web discovery supports the well-known path and robots.txt fallback, but HTML-link
  parsing and a browser package are deferred.
